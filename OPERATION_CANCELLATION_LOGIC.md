# Логика отмены операций в МЛ

## Вопрос: Должны ли меняться даты при отмене операции?

### Принятое решение: **НЕ МЕНЯТЬ ДАТЫ**

## Обоснование

### 1. Производственный учет
В реальном производстве операция может быть отменена по разным причинам:
- **Брак обнаружен после выполнения** - операция была выполнена, время потрачено, но результат не принят
- **Ошибка в документации** - операция выполнена правильно, но записана в неправильный МЛ
- **Изменение технологии** - операция выполнена, но процесс изменился

Во всех случаях **работа была выполнена**, и это нужно учитывать.

### 2. Расчет заработной платы
- Сотрудник выполнил работу → должен получить оплату
- Даже если операция отменена по административной ошибке
- Даты начала и окончания нужны для расчета отработанного времени

### 3. Аудит и отчетность
- Полная история операции: когда началась, когда завершилась, когда отменена
- Анализ эффективности: сколько времени тратится на операции, которые потом отменяются
- Выявление проблем: если много отмен после завершения → проблема в контроле качества

### 4. Юридическая защита
- Доказательство выполненной работы
- Защита от споров с сотрудниками
- Соответствие трудовому законодательству

## Текущая реализация

### Что происходит при отмене:
1. Статус меняется с "Завершена" на "Отменена"
2. В примечание добавляется: `| Отменена DD.MM.YYYY HH:mm`
3. **Даты НЕ меняются**: `ДатаНачала` и `ДатаОкончания` остаются
4. Операция остается в БД (не удаляется)

### Пример:
```
Операция: Токарная обработка
ДатаНачала: 15.03.2026 08:00
ДатаОкончания: 15.03.2026 11:30
Статус: Отменена
Примечание: Обнаружен брак | Отменена 15.03.2026 14:25
```

## Альтернативные подходы (НЕ рекомендуются)

### Вариант 1: Обнулить даты
❌ **Проблемы:**
- Потеря истории выполнения
- Невозможность рассчитать зарплату
- Нарушение аудита

### Вариант 2: Удалить операцию
❌ **Проблемы:**
- Полная потеря информации
- Невозможность анализа ошибок
- Юридические риски

### Вариант 3: Добавить поле "ДатаОтмены"
✅ **Можно рассмотреть в будущем:**
- Отдельное поле в БД для даты отмены
- Более структурированное хранение
- Сейчас дата отмены хранится в примечании - достаточно для текущих задач

## Работа с отчетами

При формировании отчетов нужно **фильтровать по статусу**:

```sql
-- Только активные операции
SELECT * FROM Операция_МЛ 
WHERE Статус IN ('В работе', 'Завершена')

-- Исключить отмененные
SELECT * FROM Операция_МЛ 
WHERE Статус != 'Отменена'

-- Анализ отмененных операций
SELECT * FROM Операция_МЛ 
WHERE Статус = 'Отменена'
```

## Бизнес-правила

1. **Можно отменить только завершенные операции**
   - Операция "В работе" → сначала завершить, потом отменить
   - Или просто не завершать, если ошибка обнаружена сразу

2. **Отмена необратима**
   - Нельзя "отменить отмену"
   - Если нужно вернуть операцию → создать новую

3. **Сохранение истории**
   - Все операции остаются в БД
   - Статус "Отменена" исключает из активных
   - Даты сохраняются для аудита

## Рекомендации для будущего

Если потребуется более детальный учет:

1. **Добавить поле `ДатаОтмены`** в модель
2. **Добавить поле `ПричинаОтмены`** (справочник причин)
3. **Добавить поле `ОтменилСотрудник`** (кто отменил)
4. **Создать таблицу истории изменений** операций

Но для текущих задач достаточно существующей реализации.
